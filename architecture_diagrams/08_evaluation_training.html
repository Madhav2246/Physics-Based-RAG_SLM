Report evaluation stages

StageScript(s)Measures
1 — Physics validationstage1_physics.py, stage1_physics_new.py0–4 physics score on golden QA for three sides: 70B (Llama golden answers, ungrounded), SYS (deployed: corpus-grounded + best-of-N), RAW (bare 0.5B, mean over samples). Seeded (cfg.SEED). The _new variant re-scores with new_checker (letter-soup fix), stores every raw answer in answers_dump.jsonl so future checkers re-score with no regeneration, writes to a separate folder (stage1_new_separate_eval/).
2 — Generation qualitystage2_generation.pyBERTScore F1, ROUGE-L, BLEU-4 vs the 70B reference; Faithfulness = cos(answer, corpus equation); Answer Relevancy = cos(answer, question). Runs entirely from stored texts — no GPU.
3 — Retrieval qualitystage3_rag.py, stage3_confirm.py, sweep_retrieval.pyHit@1 / Hit@3 (gold chunk_id containment), MRR, NDCG@3, context relevancy, full-context faithfulness. stage3_confirm re-checks the live hybrid path (dense+BM25+RRF+rerank) against anchors after the bge-large rebuild. sweep_retrieval = offline SOTA sweep over (chunk size, overlap, embedder, prefix); winner bge-large @ 384/64, Hit@3 0.65.
4 — Ablationstage4_ablation.py, stage4b_validator_test.pyComponent contribution on the same 100 QA (seed 42, stored samples): full, −reranker, −sparse, −dense, −bestofN (take sample[0]), −validator (random sample), raw_0.5b. 4b: does best-of-N selection matter at n=3 (best3 vs first3 vs rand3, corpus-grounded and raw).
5 — Token sensitivitygen_stage5_chart.py, token_sensitivity_kaggle.py, evaluation_diff_token_size/Score vs MAX_NEW_TOKENS budget; chart generation for the report.
6 — Significancestage6_significance.pyWilcoxon signed-rank (two-tailed): SYS vs 70B (all + per difficulty), SYS vs RAW, validator effect (best vs rand), best-of-N effect (best vs first). Effect size r = |Z|/√N. Produces master tables.

HITL training loop

1 · Feedback capture

Q&A page → POST /api/feedback → data/feedback/hitl_corrections.jsonl (types: correction / gap / thumbs-up; only corrections feed training).

▼

2 · train_from_feedback.py (spawned by POST /api/train/from-feedback)

  • 70/30 correction-to-base mix to prevent catastrophic forgetting; base = HuggingFaceH4/no_robots (human-written, close to Qwen-Instruct style).
  • Defaults from the API: lr 2e-4, LoRA rank 16, batch 1, 200 max steps, 10 warmup, cosine schedule; ≤3 epochs at lr 1e-4 for ~65 examples per the docstring guidance.
  • Writes adapter to models/candidate_slm (staging) — never directly to the live adapter. Live progress → data/feedback/live_training.json.
▼

3 · evaluate_candidate.py — promotion gate

Scores the candidate adapter; only on pass does it replace models/finetuned_slm (the adapter TinySLM loads at startup). Initial fine-tune: train_slm.py on data/processed/train_dataset.jsonl; train_all.ps1 at repo root drives full retraining.

Live-progress protocol (API ⇄ scripts)

synthesize_data.py      → data/evaluation/live.json
evaluate_pipeline.py    → data/evaluation/live_evaluation_nvidia_golden_qa.json
                          + eval_results_*.jsonl  (20-bin histogram source)
train_from_feedback.py  → data/feedback/live_training.json

Scripts run as subprocesses (subprocess.Popen); the API only reads these JSON files — no shared memory, crash-isolated.
