Reasoning Module — backend_new/reasoning/

Language generation only. All math is delegated to SymPy engines — the SLM never computes numbers in the deployed paths. Files: slm_model.py (class TinySLM) · prompt_builder.py (function build_prompt).

Generation — generate_multiple(prompt, n_samples, max_tokens, seed)

Chat template

Messages = fixed system prompt + user prompt, rendered via tokenizer.apply_chat_template(..., add_generation_prompt=True). System prompt (verbatim intent):

"You are a semiconductor physics assistant.
Explain the physical meaning of each symbol in an equation using a bulleted list.
Use PLAIN TEXT ONLY. Do NOT use LaTeX, \[, \(, $, or \frac.
Example output:
- Id = drain current in amperes
- Cox = oxide capacitance per unit area in F/m^2
Base your answer ONLY on the provided Evidence."
▼

Sampling (all values from utils/config.py)

  • Reproducibility (Tier 1): transformers.set_seed(cfg.SEED = 42) is called immediately before model.generate() — identical calls give identical outputs; set SEED=None for stochastic mode.
  • max_new_tokens = 128 (capped to prevent slow CPU loops) · do_sample=True · temperature = 0.3 (0.7 caused hallucinations at 0.5B) · top_p = 0.9 · num_return_sequences = n_samples = 3.
▼

Decode fix

Only newly generated tokens are decoded: output[input_length:] slice strips the prompt (the old code decoded the full sequence, so the prompt leaked into every answer). Returns list[str] of length 3.

Boundary rule (stated in code)

The SLM explains language; it never performs algebra or substitution in EXPLORE/SWEEP — those responses are composed entirely by ExplorationEngine/SweepEngine from SymPy results. LOOKUP answers are always validated afterwards by the three physics validators regardless of what the model wrote.
